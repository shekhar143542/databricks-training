# 📊 Business Pipeline & Analytics — Mini Project

An end-to-end **PySpark ETL pipeline** that cleans raw sales data, derives business metrics, segments customers, and generates a final analytical report. This project demonstrates real-world data engineering concepts including data cleaning, aggregation, customer segmentation, joins, and reporting using **Apache Spark**.

# 💡 Reflection

## Why is cleaning done before joining tables?

Joins multiply the effect of bad data. A null or duplicate key on one side of a join either drops rows unexpectedly or creates duplicate matches on the other side, causing downstream sums and counts to become incorrect. Cleaning the data first ensures accurate joins and reliable analytical results.

---

## What would go wrong if null keys are not removed?

Rows with a null `customer_id` cannot be matched reliably to a customer. In an inner join, those rows are silently dropped, resulting in undercounted activity. In an outer join, they may be grouped into an "unknown" customer, corrupting metrics such as total spend and order count.

---

## How did you decide the join order?

The pipeline begins with cleaned and deduplicated sales data keyed by `customer_id`, followed by joins with derived aggregates such as total spend, customer segmentation, and primary city. Filtering early reduces the amount of data involved in expensive joins and aggregations.

---

## Which step was most difficult and why?

Building the final reporting table was the most challenging part because a customer can place orders from multiple cities. A window function (`row_number`) was used to identify each customer's most frequently occurring city before joining all analytical results into a single summary row.

---

## How is SQL logic similar to PySpark?

PySpark's DataFrame API closely mirrors SQL concepts.

| SQL | PySpark |
|------|----------|
| WHERE | filter() |
| GROUP BY | groupBy().agg() |
| JOIN | join() |
| ORDER BY | orderBy() |
| ROW_NUMBER() OVER() | Window Functions |

The underlying logic remains the same; only the syntax differs.

---

## What challenges appear with large datasets?

- Expensive shuffles during joins and aggregations
- Data skew causing slow tasks
- Increased memory usage
- Large output files
- Partition management for efficient storage
